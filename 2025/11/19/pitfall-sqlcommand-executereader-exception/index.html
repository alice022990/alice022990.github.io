踩坑記錄：ExecuteReader() 為何沒有拋出 SqlException

Posted on 2025-11-19 In 後端開發 , 踩坑記錄
記錄 SqlCommand.ExecuteReader() 沒有如預期拋出 SqlException 的問題，以及解決方案與原因分析。

最近在測試 Stored Procedure 的錯誤處理時，發現一個奇怪的現象：
明明 SP 內部發生了錯誤並執行了 RAISERROR，但 C# 後端卻沒有捕捉到任何異常，甚至還記錄了「作業執行成功」的日誌。
查了一下才發現，問題出在 ExecuteReader()

背景說明

下面重現我本來寫的 Stored Procedure 跟 C#後端

Stored Procedure 內容

這是一個簡化版的 SP，使用了 TRY…CATCH 來包 transaction。在 TRY 區塊中，我故意放了一個 SELECT 1/0 來觸發錯誤。因此在 CATCH 區塊中，會 Rollback 事務，並使用 RAISERROR 拋出錯誤訊息。

  • 錯誤沒有用 throw，是因為工作用的資料庫相容性等級太低，用不了

C# 後端

這是一個簡單的主控台應用程式，使用了 cmd.ExecuteReader() 來執行 SP，並期望 SqlException 會在 catch (SqlException sqlEx) 區塊被捕捉，記錄到 Error Log 中。
但執行完的結果只有出現 Info Log，甚至記錄到了 「=== 作業執行成功 ===」，try-catch 區塊完全沒有捕捉到任何 SqlException。

為什麼後端沒有捕捉到錯誤？

RAISERROR 的行為

當 SP 執行到 RAISERROR 時，SQL Server：

關鍵：RAISERROR 的嚴重性級別，會影響錯誤是否會中斷連線

  • 觸發的錯誤是@ErrorSeverity = 16（一般錯誤）
  • 這個級別的錯誤不會立即中斷 SQL 批次
  • SP 會正常完成執行，只是返回一個錯誤訊息

C# ExecuteReader() 的行為

ExecuteReader() 實際上完成了以下步驟：

  1. 發送命令到 SQL Server

  2. SQL Server 執行 SP

  3. SP 內部：

    • 執行 SELECT 1/0 → 錯誤
    • 進入 CATCH 區塊
    • 執行 ROLLBACK
    • 執行 RAISERROR → 產生錯誤訊息
    • SP 執行完畢（沒有崩潰）

  4. SQL Server 返回結果給 C#：

    • 空的結果集（因為 SELECT @DeletedRows... 沒成功執行）
    • 有一個錯誤訊息（來自 RAISERROR）

  5. C# 的 SqlDataReader：

    執行流程：

    1. ExecuteReader() 打開結果流
    2. reader.Read() 嘗試讀取第一行 → 返回 false（沒有資料）
    3. 不進入 if 區塊
    4. reader 被 dispose，結果流被提前關閉
    5. 錯誤訊息還在流中，但沒被讀取！

為什麼沒有拋出 SqlException？

ExecuteReader() 只有在以下情況才會拋出異常：

  1. 嚴重錯誤（Severity >= 20）

  2. 連線層級的錯誤：例如：語法錯誤、權限錯誤、連線中斷

  3. SQL 錯誤發生在 TRY…CATCH 之外：錯誤沒有被捕捉，會直接拋給 C#

小結

程式沒有捕捉到錯誤是因為：

  1. 錯誤被 SP 內部的 TRY...CATCH 捕捉了
  2. RAISERROR(Severity=16) 不會中斷 SP 執行
  3. SP 正常完成，只是沒有返回資料
  4. ExecuteReader() 看到「沒有結果集」，不會拋出異常
  5. 本來的程式只檢查了 reader.Read()，返回 false 就結束了

解法二：保持使用 ExecuteReader，但檢查 NextResult

因為 SQL Server 會透過 TDS（Tabular Data Stream）協定以串流方式將結果傳回給用戶端，而 結果集的資料列 會先於 例外訊息。

因此，用戶端 API 必須先把前面的資料全部讀取完，才能看到並拋出該錯誤。

利用 NextResult() 迴圈可以確保 TDS 解析器已經完整讀取整個結果集。
