搞懂加密、雜湊、加鹽與 ASP.NET Core 密碼加密

Posted on 2025-11-15 In 後端開發 , 資訊安全
深入解析密碼儲存的安全機制，包含 Hash、Salt、Encoding、Encryption 的差異，以及 ASP.NET Core PasswordHasher 的實作原理與使用方式。

剛開始學習密碼儲存時，我一直搞不清楚這些名詞：

  • 密碼要「編碼」還是「加密」？
  • 什麼是「雜湊」？跟加密有什麼不同？
  • 為什麼還要「加鹽」？鹽又是什麼？

這篇文章會分享我的學習心得，從最基礎的概念開始，一步步帶你了解：

  • 編碼、加密、雜湊、加鹽是什麼
  • 在 ASP.NET Core 如何安全地處理密碼

相關名詞介紹

什麼是編碼（Encoding）

編碼是把資料換一種表示方式，讓資料能安全傳輸或顯示。但內容不會變、可回推原本的資料。
常見的例子：

  1. Base64 編碼：把二進位資料轉成可安全傳輸的字串。
  2. URL Encoding：把網址中的特殊字元（空白、標點符號等等）編碼成符合 URL 的格式，讓網址能安全傳輸。

什麼是加密（Encryption）

加密是把明文結合加密演算法 + 密鑰，轉成密文，只要有密鑰就可以還原明文。

  • 因為「可解密」，不適合直接用來儲存密碼。
  • 攻擊者只要掌握加密方法跟拿到密鑰，就能還原所有密碼。

什麼是雜湊 （Hash）

雜湊是將資料，透過雜湊函式（hash function）將密碼轉換成一段不可逆的固定長度字串，無法從轉換後的 Hash 字串，去推導還原出原本的模樣(原密碼)。
同樣的輸入經過雜湊，一定會得到一樣的結果，所以在驗證登入的時候，只要對輸入的密碼進行一樣的雜湊方法，跟資料庫裡面存的 Hash 字串比對是否相同就好。

特性 說明
不可逆 無法從 Hash 字串還原原始密碼
固定長度 不管你輸入多長，輸出都是固定長度（例如 SHA-256 固定 64 字）
要用於比對 通常用來比對是否一致，而不是還原

什麼是加鹽 （Salt）

因為「同樣的密碼」會產生「同樣的 hash」，如果有一張表（彩虹表）記錄明文跟其對應的雜湊值，那駭客得到雜湊值後可以快速查出對應的密碼。
為了避免這種狀況，可以對密碼進行「加鹽」，提高暴力破解難度。
加鹽指的是在密碼中任意位置，加上一段隨機字串，再做雜湊。

加鹽後

即使兩人密碼相同都是 123456，但系統產生不同 Salt，最終的 hash 依然不同：

密碼 Salt 輸入給 Hash Function Hash
123456 X7@1 Hash(123456, X7@1) AAA23
123456 K!9Z Hash(123456, K!9Z) CCB89

即使密碼相同，在沒有取得鹽，也不知道鹽是加在密碼的哪個位置的情形下，駭客也難以在短時間內破解密碼。

ASP.NET Core 怎麼安全儲存密碼？

在 ASP.NET Core，可以利用 Identity library 中的 PasswordHasher<TUser> 類別來處理密碼雜湊跟驗證

PasswordHasher<TUser>

核心概念與功能

主要方法：

  1. HashPassword(TUser user, string password)：雜湊密碼

    • 不僅將使用者提供的純文字密碼轉換成密碼的雜湊值，還將所有必要的資訊打包成一個單一的字串（包含雜湊值、鹽值和迭代次數等資訊），儲存在資料庫中。
      例如：版本識別:迭代次數:鹽值:雜湊結果
    • 過程：
      1. PasswordHasher 內部會生成一個隨機且唯一的鹽值（Salt）。
      2. 它會使用這個鹽值和密碼進行 PBKDF2 運算，得到最終的雜湊結果。
      3. 最後，它會將鹽值本身連同雜湊結果及其他資訊（如迭代次數）組合起來，形成一個字串（通常以 Base64 編碼）。

  2. VerifyHashedPassword(TUser user, string hashedPassword,
    string providedPassword)：驗證使用者密碼

    • 比較使用者輸入的密碼（providedPassword）是否與儲存在資料庫中的雜湊密碼（hashedPassword）匹配。
    • 過程：
      1. 解析字串: PasswordHasher 會接收儲存的 hashedPassword 字串。它會解析這個複合字串，從中提取出：
        • 當初用於雜湊的鹽值。
        • 當初使用的迭代次數（Iteration Count）和其他配置參數。
        • 儲存的最終雜湊結果。
      2. 重新雜湊: 它會使用提取出來的鹽值和使用者這次輸入的純文字密碼（providedPassword），執行完全相同的雜湊演算法（如 PBKDF2）和迭代次數。
      3. 比較結果: 得到新的雜湊結果後，它會將這個新生成的雜湊結果與儲存的最終雜湊結果進行位元組（byte）級別的比較。
      4. 返回結果: 如果兩者完全匹配，則驗證成功。
    • 驗證密碼，傳回的結果會是下面三個其中之一
      • PasswordVerificationResult.Success：密碼正確
      • PasswordVerificationResult.SuccessRehashNeeded：密碼正確，但我們需要將雜湊演算法更新為新的演算法。
      • PasswordVerificationResult.Failed：密碼不正確
特點
  • 實作IPasswordHasher<TUser> 介面
  • 安全機制
    • 使用 PBKDF2（Password-Based Key Derivation Function 2）演算法配合 HMAC-SHA256/SHA512
    • 執行多次迭代（重複雜湊過程），.NET Core 預設是 10,000 次，使得暴力破解（Brute-Force）攻擊的速度極慢，增加了攻擊者的成本
  • 客製化
    • 可以透過設定 IdentityOptions 來調整預設的雜湊行為
    • 可以建立一個自訂的類別來實作 IPasswordHasher<TUser> 介面，然後將其註冊到依賴注入容器中，替換掉預設的 PasswordHasher

學的時候的疑惑

  • 為什麼鹽存在資料庫沒有安全問題？
    • 因為就算攻擊者知道 Salt，只要 Salt 是每個密碼不同的隨機值，攻擊者無法一次破解很多帳號，也無法使用彩虹表
    • Salt 的作用：
      • 增加破解成本
      • 讓每個密碼的 Hash 都唯一
      • 讓攻擊者沒辦法用事先算好的表來對照
  • 就算攻擊者知道 Hash（雜湊值）、Salt、使用的演算法（如 PBKDF2），也很難破解密碼，因為
    • Hash 是不可逆的，能做的只有「暴力嘗試」，對所有可能密碼做：Hash(猜的密碼 + 已知的鹽)，然後比對結果，這對於強密碼來說計算成本極高。
    • 並且迭代（PBKDF2 演算法中重複執行雜湊計算的次數）也會增加計算成本
